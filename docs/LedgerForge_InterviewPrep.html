Operations team needs a daily report with total transaction amount and count per branch, along with branch name and state.
Task

Join transaction → account → branch to get branch details for each transaction.

Aggregate: total amount, transaction count, average amount per branch. Sort by total amount descending.

Output

Approach
// Filter early to reduce data volume before joins
val txns = transaction
  .filter(col("status") =!= "Reversed")
  .filter(col("account_id").isNotNull && col("account_id") =!= -1)

val result = txns
  .join(account.filter(col("branch_id").isNotNull), "account_id")
  .join(broadcast(branch), "branch_id") // branch is 500 rows — broadcast!
  .groupBy("branch_id", "branch_name", "state")
  .agg(

    count("*").as("txn_count"),
    avg("amount").as("avg_amount")

Context
Every transaction must be enriched with product name and group. Product table is 5K rows vs 50M transaction rows — perfect broadcast candidate.
Task

Run 1: Join without any hint. Capture physical plan with explain(true). Record shuffle stages and total task count.

Run 2: Use broadcast(product) hint. Capture plan again. Compare: shuffle stages, task count, execution time.

Approach
// Run 1: No hint — likely SortMergeJoin with shuffle
val naiveJoin = transaction.join(product, Seq("product_id"), "left")
naiveJoin.explain(true)

// Run 2: Broadcast hint — no shuffle needed
import org.apache.spark.sql.functions.broadcast
val enriched = transaction.join(broadcast(product), Seq("product_id"), "left")

enriched.explain(true) // Look for BroadcastHashJoin — no Exchange node

Regulatory reporting requires the customer's risk tier as it was on the date of each transaction — not the current value. A transaction on Jan 15, 2026 needs Alice's risk tier as of Jan 15, not today.
Task

Join: transaction → account (get customer_id) → customer_scd2

The SCD2 join condition: event_date >= effective_start AND event_date <= coalesce(effective_end, '9999-12-31')

Measure execution time with full SCD2 join vs. incorrect fast path (is_current=true only).

Output
transaction_idevent_dateamountcustomer_idfirst_namelast_namerisk_tierversion
Approach
val scd2 = customerScd2.withColumn("eff_end_safe",
  coalesce(col("effective_end"), lit("9999-12-31").cast(DateType)))

  .join(account, "account_id")   // get customer_id
  .join(scd2,

Data Quality Quarantine PipelineMEDIUM
6 quality rules · rejection_reasons array · accumulators
▼

Quality Rules

1. transaction_id IS NOT NULL

2. account_id IS NOT NULL AND account_id > 0

3. amount IS NOT NULL AND amount != -999999.99

4. currency IS NOT NULL AND currency != "XXX"

5. event_date IS NOT NULL

6. transaction_type IS NOT NULL AND transaction_type != "UNKNOWN"

Approach
// Build rejection_reasons as an array of strings
val withReasons = transaction.withColumn("rejection_reasons",
  array(
    when(col("transaction_id").isNull, lit("NULL_TXN_ID")),
    when(col("account_id").isNull || col("account_id") <= 0, lit("INVALID_ACCOUNT")),
    when(col("amount").isNull || col("amount") === -999999.99, lit("INVALID_AMOUNT")),
    when(col("currency").isNull || col("currency") === "XXX", lit("INVALID_CURRENCY")),
    when(col("event_date").isNull, lit("NULL_DATE")),
    when(col("transaction_type").isNull || col("transaction_type") === "UNKNOWN", lit("UNKNOWN_TYPE"))
  ).cast("array<string>")  // remove nulls from array
)
val valid = withReasons.filter(size(col("rejection_reasons")) === 0)
val quarantine = withReasons.filter(size(col("rejection_reasons")) > 0)
💡Use array_compact() (Spark 3.4+) or array_remove(arr, null) to clean nulls from the rejection_reasons array. Use LongAccumulator inside foreachPartition to count violations without triggering extra scans.

UDF vs. Native — Risk Bucket ClassificationHARD

▼

Risk Rules

Why Native Wins

UDF: Spark serializes each Row to Java objects, calls your Scala function, then deserializes back. Bypasses Catalyst — no predicate pushdown, no constant folding, no WholeStageCodegen fusion. Every row crosses the Spark→JVM boundary.

Native when/otherwise: Catalyst compiles the expression tree to bytecode via WholeStageCodegen. The entire chain of conditions runs in a single fused loop without object creation or serialization. Typically 3–10× faster on large datasets.

// Native — Catalyst can optimize and codegen this
val nativeResult = transaction.withColumn("risk_bucket",
  when(col("amount").isNull || col("transaction_type").isNull, "UNKNOWN")
  .when(col("amount") > 5000, "CRITICAL")
  .when(col("amount") > 1000 && col("transaction_type") === "Transfer", "HIGH")
  .when(col("amount") > 500, "MEDIUM")
  .otherwise("LOW"))
⚠When you must use a UDF (complex external library calls, stateful logic), consider pandas_udf (Arrow-based vectorized UDF) which processes batches instead of row-by-row — typically 10–100× faster than standard UDFs.

Partition Optimization — Small File ProblemMEDIUM
repartition · coalesce · inputFiles · controlled file count
▼
repartitioncoalescesmall file probleminputFiles
Key Insight

coalesce(4) alone after reading a large dataset may not work as expected — coalesce is a narrow transformation that avoids shuffles but can only reduce partitions within an existing partition distribution. If the data is not already partitioned by event_date, coalescing may group rows from different dates together.

The correct approach: use repartition(4, col("event_date")) which forces a shuffle that guarantees exactly 4 output files per date partition.

df.repartition(4, col("event_date")) // ✅ correct — shuffles to exactly 4 per date
  .write.partitionBy("event_date").parquet(outputPath)
P11
Salting for Skewed JoinsHARD
Zipf skew · salt + explode technique · task variance
▼
data skewsalting patternexplodeSpark UI task metrics
The Problem
account_id follows Zipf distribution — top 1% of accounts have 30%+ of all transactions. Joining transaction × account on account_id creates skewed shuffle partitions where a few tasks run 100× longer than others.
Salting Solution
val N = 10 // salt range

// Add random salt to transactions
val txSalted = transaction
  .withColumn("salt", (rand() * N).cast(IntegerType))

// Explode account to N copies, each with a different salt
val accExploded = account
  .withColumn("salt", explode(array((0 until N).map(lit): _*)))

// Join on (account_id, salt) — distributes hot accounts across N partitions
val result = txSalted
  .join(accExploded, Seq("account_id", "salt"))
  .drop("salt")
💡Salting increases account table size by N×. Higher N = better skew reduction but more memory for the exploded dimension. N=10 reduces max task time to ~1/10 of unsalted. AQE's skew join optimization (Spark 3+) can handle this automatically without manual salting.

▼

Key Concept

spark.sql.functions.expr() parses a string as a Spark SQL expression — enabling config-defined column logic without code changes.

// Config defines: { name = "full_name", expr = "concat(first_name, ' ', last_name)" }
def transform(df: DataFrame, config: Config): DataFrame = {
  val withAdded = config.getConfigList("add_columns").asScala.foldLeft(df) {
    (acc, col) => acc.withColumn(col.getString("name"), expr(col.getString("expr")))
  }
  // apply renames, drops similarly...
  withAdded
}
⚠expr() evaluates at runtime — expression errors surface as runtime exceptions, not compile-time errors. Add validation in tests that verify every configured expression can be parsed and evaluated on sample data.

stream.writeStream
  .foreachBatch { (batchDF: DataFrame, batchId: Long) =>
    val valid = batchDF.filter(col("amount").isNotNull && col("amount") =!= -999999.99)
    val quarantine = batchDF.filter(col("amount").isNull || col("amount") === -999999.99)

    // Idempotent: overwrite same path per batchId on retry
    valid.write.mode("overwrite").parquet(s"output/streaming/valid/batch=$batchId")
    quarantine.write.mode("overwrite").parquet(s"output/streaming/quarantine/batch=$batchId")
  }
  .option("checkpointLocation", "output/streaming/dq_checkpoint")
  .start()
💡foreachBatch gives you a regular DataFrame — you can use any DataFrame API including multiple writes, joins with static data, Delta MERGE operations, or custom sinks. The batchId is monotonically increasing and can serve as an idempotency key for the sink.

Code
val metrics = stream
  .groupBy("transaction_type", "currency")
  .agg(sum("amount").as("total"), count("*").as("volume"))

metrics.writeStream
  .outputMode("complete") // rewrites entire 20-row result each batch
  .format("memory")
  .queryName("dashboard")
  .start()

// Query at any time
spark.sql("SELECT * FROM dashboard ORDER BY total DESC").show()
⚠complete mode requires keeping ALL aggregation state in memory forever (no watermark support). Only feasible when result cardinality is small (here: 4 types × 5 currencies = 20 rows). Never use complete mode on high-cardinality groupings like account_id — use update mode instead.
