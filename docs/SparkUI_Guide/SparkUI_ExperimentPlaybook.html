SECTION 01
How to Observe Spark UI While Running Experiments

The key insight: open Spark UI before the job starts, not after it finishes. Most experiments complete in minutes — if you only look after completion, you miss the live execution view, straggler detection, and real-time task distribution. Open four browser tabs pre-loaded with the four most useful views before you run a single line of code.

⚠️ Spark UI does not auto-refresh. Manually refresh every 10–30 seconds during execution. Set a timer if needed — missing the peak of a skewed stage means missing the most important diagnostic moment.
Pre-load These 4 Browser Tabs Before Every Experiment
1
localhost:4040/jobs/
Watch active jobs. Count stages, spot stalling progress bars, confirm you're watching the right job.
2
localhost:4040/stages/
Watch active stages and live task distributions. Click into active stages to see the real-time task histogram.
3
localhost:4040/SQL/
Inspect physical plans. Populates as soon as Spark submits the query — before tasks start running.
4
localhost:4040/executors/
Watch resource usage, GC time, shuffle imbalance, and dead executors in real time.
What to Track Live in Each Tab
📋
Jobs Tab LIVE
/jobs/
→Stages Succeeded/Total — e.g. 2/4 means 2 done, 2 in progress. Count stages before and after optimization to confirm shuffles reduced.
→Task progress bar stalling at 90%+ — if the bar stays frozen near the end for a long time, one or a few tasks are stragglers. Navigate immediately to Stages tab.
→Job description link — links to source file + line number. Use to confirm you're watching baseline vs. optimized run.
→Live signal — pattern like 180/423 (9 running) means 9 tasks running while 180 are done. If those 9 have been running much longer than others → skew.
📊
Stages Tab LIVE
/stages/
→Task duration column — sort descending. If max duration is 10× median, you have skew. This is your most important metric.
→Shuffle Read Size column — sort descending. If one task reads far more bytes than others, that's the skewed key.
→Locality Level column — PROCESS_LOCAL (fastest) vs. ANY (cross-rack network fetch). High ANY = scheduler couldn't place tasks near data.
→GC Time column — sort descending. Task with GC > 10% of duration is under memory pressure.
🗺️
SQL Tab
/SQL/ — populates before tasks start
→Count Exchange nodes immediately. Baseline should have more; optimized should have fewer or different types.
→Find join node type. SortMergeJoin vs. BroadcastHashJoin is the most critical difference to find.
→FileScan node. Look for PartitionFilters in scan details. If empty, partition pruning is broken.
→AdaptiveSparkPlan wrapper. If present, AQE is active — plan may change between start and completion. Always read the final executed plan.
→Row count estimates. If wildly wrong (50M estimated as 100), optimizer is flying blind — statistics are missing.
🖥️
Executors Tab LIVE
/executors/ — check every minute
→GC Time column — should show GC as a small fraction of total task time. High = heap pressure.
→Shuffle Read/Write per executor — should be roughly balanced. Large imbalance = data skew confirmed at executor level.
→Failed Tasks column — any value > 0 warrants investigation. Click the executor ID link to access executor logs.
→Storage Memory Used/Total — if at 100%, executor is under memory pressure and cached DataFrames may be evicted mid-job.
🗄️ Storage Tab — For Caching Experiments
• Before .cache(): tab is empty.
• After first action that triggers caching: a new entry appears with storage level, partition count, and size.
• Fraction Cached must reach 1.0 (100%) for the cache to be fully effective. Any value below 1.0 = eviction happening.
• Compare Size in Memory between MEMORY_ONLY, MEMORY_AND_DISK, and DISK_ONLY storage levels to measure serialization overhead.

EXP 01
Skewed Joins — Salting
Naive join on account_id vs. salted join to distribute hot keys
▼
AspectBASELINE Naive JoinOPTIMIZED Salted Join
Join typeSortMergeJoin on account_idSortMergeJoin on (salt, account_id)
Task distributionExtreme right tail — 1 massive stragglerRoughly even histogram
Max task durationMinutes (10–100× median)Similar to median task time
Shuffle Read max/median ratio> 100× (one task reads most data)< 5× (uniform distribution)
Physical planSimple join node on account_idExtra Project (salt), join on (salt, account_id), salt drop
Stage countN stagesN+1 stages (extra for salt explode on small side)
Step-by-Step Observation
1In the baseline: click into the join stage → sort tasks by Shuffle Read Size descending → observe the top task reading orders of magnitude more data than the median. This is your hot key.
2Look at the task duration histogram — confirm the right tail. One bar should extend far beyond the cluster of normal tasks.
3In the optimized: find the additional pre-join Project stage in the SQL plan that adds the salt column. This is the overhead of the technique.
4The join stage in the optimized run should show a uniform task duration histogram — no right tail. Max/median ratio should be <3×.
5The optimized plan will have one extra stage (for the salt explode/replicate on the small side), but each stage will have balanced tasks. Total time should still be faster despite the extra stage.
6Bonus: Check the Executors tab during baseline run — one executor should show far higher Shuffle Read than others. In the optimized run, shuffle should be balanced across all executors.
💡 AQE's automatic skew join optimization (spark.sql.adaptive.skewJoin.enabled = true) can handle this without manual salting. Run the experiment a third time with AQE enabled — compare all three plans side by side. AQE-handled skew shows SkewJoin=true on the SortMergeJoin node.
EXP 02
Broadcast Joins
SortMergeJoin vs. BroadcastHashJoin for small dimension tables
▼
AspectBASELINE SortMergeJoinOPTIMIZED BroadcastHashJoin
Exchange nodes2 (one per join side)1 BroadcastExchange (small side only)
Stage count3 (two shuffle stages + join)2 (scan/filter + join)
Shuffle Write (large side)Hundreds of MB0 — large side never shuffled
Join node in SQL tabSortMergeJoinBroadcastHashJoin
Sort nodes in planYes (before each SMJ input)None — no sorting needed for hash probe
Step-by-Step Observation
1Baseline SQL tab: find two Exchange hashpartitioning(...) nodes feeding a Sort feeding SortMergeJoin. Count them — this is the shuffle tax you're eliminating.
2Optimized SQL tab: find BroadcastExchange on the small side only feeding BroadcastHashJoin. Crucially, there is no Exchange node for the large side — this is the entire speedup mechanism.
3In the Stages tab: the baseline has a shuffle stage for the large table with significant Shuffle Write. In the optimized run, this stage disappears entirely.
4Confirm in Jobs tab: the optimized job should show fewer stages total (Stages: Succeeded/Total).
5If BroadcastHashJoin is not appearing when expected: check spark.sql.autoBroadcastJoinThreshold in the Environment tab. If -1, broadcasting is disabled. Also check if statistics are computed (ANALYZE TABLE).
💡 The Environment tab (/environment/) is the ground truth for all Spark configs. Always verify autoBroadcastJoinThreshold here — if it's lower than your small table's actual size, broadcasts are silently disabled even without setting it to -1.
EXP 03
Partition Pruning
Full table scan vs. reading only filtered event_date partitions
▼
AspectBASELINE Full ScanOPTIMIZED Pruned
FileScan partitions read30 (all 30 date partitions)1–3 (only filtered dates)
Input size (Stages tab)Full 50M row datasetSmall fraction — 1–3 day subset
PartitionFilters in SQL tab[] (empty — not pushed down)[isnotnull(event_date), (event_date = '2024-01-15')]
Stage durationLong — reading unnecessary dataShort — minimal I/O
Records readAll 50M rows~1.6M rows (1/30th)
Step-by-Step Observation
1In the SQL tab, click the FileScan node for the baseline. Look at the PartitionFilters field — it should be empty [], confirming Spark is reading all 30 date partitions.
2In the Stages tab, look at the Input column for the scan stage — it should show the full dataset size.
3In the optimized SQL tab, click the FileScan node — PartitionFilters should now show your date filter. The PushedFilters field should also be populated for row-level predicates.
4Note: PushedFilters (row-level, inside the Parquet reader) is different from PartitionFilters (directory-level). Both should be populated in a fully optimized plan.
5Common failure mode: year(event_date) = 2024 will NOT trigger partition pruning because Spark can't infer which directories to skip from a UDF call. Use event_date >= '2024-01-01' AND event_date < '2025-01-01' instead.
⚠️ Dynamic Partition Pruning (DPP) is different from static partition pruning. DPP triggers when a filter value comes from a join result (e.g., filter on a fact table based on a dimension value). DPP requires AQE and a specific join pattern — check the SQL plan for a DynamicPruningExpression node when testing DPP scenarios.
EXP 04
Caching Strategies
No cache vs. cache() vs. persist() vs. checkpoint() — 3 consumers
▼
AspectNO CACHEWITH CACHE
Job count (for 3 consumers)3 × full DAG recomputed1 full DAG + 2 cheap reads from cache
Storage tab entriesEmpty1 entry per cached DataFrame
Job duration (2nd and 3rd)Same as 1st job — full recomputeVery fast — reads from memory/disk
Plan for cached queriesFull FileScan + Filter + ... DAGInMemoryTableScan (single fast node)
Stage count per consumerFull pipeline stages repeated1–2 stages — just reads cached data
Step-by-Step Observation
1Without cache: run and count jobs in the Jobs tab. Three separate queries each show a full DAG — all the same expensive stages replicated 3 times.
2With .cache(): after the first action, go to the Storage tab — confirm an entry appears with Fraction Cached = 1.0. If it shows <1.0, some partitions were evicted.
3The 2nd and 3rd jobs should complete in seconds with far fewer stages. In the SQL tab, the physical plan for cached queries shows InMemoryTableScan instead of the original complex DAG.
4Compare MEMORY_ONLY vs. DISK_ONLY: the Storage tab shows Size in Memory = 0 for DISK_ONLY. Stage durations for the 2nd/3rd consumers will be longer due to disk I/O instead of RAM reads.
5Checkpoint experiment: unlike cache/persist, checkpoint is eager — it immediately triggers computation and writes to HDFS/S3. Check that the checkpoint write appears as a separate job in the Jobs tab. After checkpointing, the lineage graph in the SQL tab should be truncated — the plan starts at the checkpoint, not the original scan.
💡 Call .unpersist() between experiments to clean up cached DataFrames. Check the Storage tab after unpersist to confirm the entry disappears. Stale cached DataFrames across experiments contaminate your measurements by holding executor memory that other DataFrames need.
EXP 05
Adaptive Query Execution (AQE)
AQE off vs. AQE on vs. AQE + skew optimization
▼
AspectAQE OFFAQE ON
Physical plan root nodeStandard physical planAdaptiveSparkPlan wrapper
Post-shuffle partition countAlways equals shuffle.partitions (e.g. 200)Fewer — coalesced small partitions (e.g. 12)
Join type at runtimeStays SortMergeJoin even if build side shrinksMay upgrade to BroadcastHashJoin at runtime
Skewed join handlingOne massive straggler taskMultiple balanced tasks (partition split)
Post-shuffle plan nodeExchangeAQEShuffleRead (coalesced)
Step-by-Step Observation
1AQE Off: SQL tab shows a static physical plan with fixed Exchange nodes. Stages tab shows exactly shuffle.partitions (e.g., 200) tasks in every post-shuffle stage — regardless of actual data size.
2AQE On: SQL tab shows AdaptiveSparkPlan at the root. After execution, expand the "Final Plan" section and compare it to the initial plan — they will likely differ.
3Look for AQEShuffleRead in the final plan — this replaces Exchange in the reduced-partition stages. Count tasks in the post-shuffle stage: expect a dramatic reduction (e.g., 200 → 15 tasks for a small dataset).
4Skew sub-experiment (skewJoin.enabled = true): look for SkewJoin=true annotation on the SortMergeJoin node. Observe more tasks than shuffle.partitions in that stage — because the skewed partition was split into sub-tasks.
5Track the key metric: Shuffle Read Size (median per task). With AQE coalescing, fewer tasks each process more data — median should be closer to the 100–200MB target. Without AQE, 200 tasks each processing 500KB means 99% scheduling overhead.
EXP 06
Deduplication Strategies
dropDuplicates vs. row_number() window vs. struct-max aggregation
▼
AspectdropDuplicatesrow_number() WindowStruct-max Agg
Physical planSort + Exchange + HashAggregateExchange + Sort + WindowExchange + HashAggregate
DeterminismNon-deterministic (any row kept)Deterministic (specific row by order)Deterministic
Shuffle volumeFull dataset shuffledFull dataset shuffledReduced (partial agg before shuffle)
SQL plan nodeNo Window nodeWindow node presentTwo HashAggregate nodes
Step-by-Step Observation
1In the SQL tab, look for a Window node in the row_number() plan. This confirms the window function is being used. The window plan shows a Sort inside the window for ordering — extra CPU overhead vs. aggregation.
2Compare shuffle sizes between approaches in the Stages tab — the struct-max aggregation may show smaller Shuffle Write because aggregation can happen partially before the shuffle (look for two HashAggregate nodes: one pre-shuffle, one post-shuffle).
3For production datasets with ~2% duplicate rate, the struct-max approach should show the smallest shuffle write in the Stages tab. More data moves in the window/dropDuplicates approaches because no pre-shuffle reduction is possible.
4Critical for production: dropDuplicates("transaction_id") keeps an arbitrary row from each duplicate group — verify this matches your business requirement. If you need to keep the latest row (by ingest_date), only row_number() with explicit ordering guarantees this.
EXP 07
Shuffle Partition Tuning
Default 200 vs. tuned N vs. AQE auto-coalesce
▼
Aspect200 partitions (default)Tuned NAQE auto-coalesce
Task count in shuffle stages200 (fixed)N (your tuned value)Varies — auto-adapted, often <20
Per-task shuffle read sizeTiny if data is small (e.g. 500KB)Target 100–200MBAuto-adjusted to ~1MB+ minimum
Task overheadHigh — 200 task launches for tiny dataLower — right-sized tasksLowest — minimal tasks needed
Plan node for shuffle readExchangeExchangeAQEShuffleRead
Step-by-Step Observation
1With 200 partitions on a small dataset: Stages tab shows 200 tasks per shuffle stage, each processing maybe 200–500KB. Stage duration is dominated by task scheduling overhead, not computation. The task histogram will show 200 tiny fast bars.
2With tuned N: fewer tasks, each processing more data. Stage duration decreases significantly. Find the optimal N using: N ≈ total_shuffle_write_bytes / (128 * 1024 * 1024). Check shuffle write in Stages tab to get total shuffle write size.
3With AQE: SQL tab shows AQEShuffleRead and the number of tasks adapts automatically. Check the final coalesced partition count — it should be far below 200 for small datasets.
4Key metric to track: Shuffle Read Size (median per task). Target 100–200MB per task. Use this to back-calculate the right spark.sql.shuffle.partitions value for your specific data size.
💡 The formula: optimal partition count ≈ total_shuffle_write / 128MB. Example: if Stages tab shows 12.8GB total shuffle write → set shuffle.partitions = 100. Enable AQE to let Spark auto-tune from there. Never blindly use the default 200 in production — it's tuned for clusters, not local mode with small data.
EXP 08
UDF vs. Native Expressions
Scala UDF breaks WholeStageCodegen — native when/otherwise stays inside codegen
▼
AspectScala UDFNative when/otherwise
WholeStageCodegenBroken — UDF is opaque to CatalystIntact — fused into one codegen block
Physical plan signatureDeserializeToObject → MapPartitions → SerializeFromObject*(N) Project [...CASE WHEN...] — inside codegen
CPU time per taskHigher — boxing/unboxing, object allocationLower — tight generated loop, no objects
GC timeHigher — intermediate JVM objects per rowLower — no row-level object creation
Stage durationLonger for same record countShorter — 3–10× improvement typical
Step-by-Step Observation
1In the SQL tab, look at the physical plan text (use .explain("formatted") for cleaner output). UDF plan: the UDF computation appears outside WholeStageCodegen brackets. You will see DeserializeToObject, MapPartitions, SerializeFromObject — these are the boundary-crossing nodes.
2Optimized plan: the CASE WHEN ... THEN ... logic is fused inside *(N) Project [...]. The entire operation is in one codegen block — look for the asterisk * prefix indicating JIT compilation.
3In the Stages tab, compare GC time between the two approaches. The UDF version will show higher GC time per task due to boxing/unboxing and intermediate object allocation.
4Also compare CPU time per task — the native expression version should have lower CPU time for the same record count. On 50M rows, this difference can be 30–60 seconds of total CPU time saved.
5The lesson: even a "simple" UDF like (x: Double) => x * 2 breaks codegen for everything downstream, not just its own operation. Replace with col("amount") * 2 and the entire pipeline stays in one fused codegen block.
SECTION 03
Spark UI Investigation Workflow

Use this repeatable 7-step workflow for every experiment — both the baseline and the optimized run. Apply it consistently and you'll build muscle memory for Spark debugging.

1
SQL Tab → Find the query → Read the physical plan
This is always your first move. The SQL tab populates before tasks even start. Identify the plan structure before looking at any runtime metrics. Use .explain("formatted") in code for a cleaner text version alongside the UI view.
2
Count Exchange nodes
Baseline has N exchanges → optimized should have M exchanges where M < N. Record the count. Each Exchange is a stage boundary, a shuffle, and a network round trip. This single number tells you how optimized the plan is.
3
Identify join strategies
Is it SMJ or BHJ? Is there a CartesianProduct (bad — check for missing join conditions)? Is BroadcastExchange on the correct small side? For each join node, ask: could this be broadcast? Should this be bucketed to eliminate the shuffle entirely?
4
Check scan nodes
Are PartitionFilters populated? Are PushedFilters populated? Are only the needed columns listed in the schema (column pruning)? If PartitionFilters is empty and you filtered on a partition column, your filter isn't being pushed down.
5
Stages tab → Slowest stage → Click into it
Sort by Duration descending. Click into the worst stage. Look at the task duration histogram — narrow bell (healthy), right tail (skew). Check Shuffle Read Size per task (sort descending). Is there spill? High GC? These three metrics tell you the root cause of slowness.
6
Executors tab → Systemic health check
Is GC time reasonable across executors? Are shuffle read/write balanced? Are any executors dead or degraded? Task counts should be roughly proportional to core count. Any executor doing 10× more work than others is a skew or scheduler problem.
7
Compare key metrics between baseline and optimized
Record all metrics in your comparison table. Confirm the plan changed as expected (not just that it's faster — structural correctness matters for scaling). A lucky 2× speedup from unchanged plan won't scale. A structurally correct plan with fewer exchanges will.
Key Metrics to Track Every Experiment
• Total job duration (wall time)
• Slowest stage duration
• Total shuffle write size
• Max task duration / Median task duration (skew ratio)
• Number of stages
• Exchange node count in physical plan
• Partition count in post-shuffle stages
• Spill (Disk) — should be 0
💡 Document your findings in a table for every experiment. Empirical before/after records are far more valuable than intuition. When asked "why is the optimized version faster?", you should be able to answer with specific numbers, not just "because broadcast joins are faster."
SECTION 04
Spark UI Signals That Indicate Problems

These are the red flags you'll see repeatedly across experiments and in production. Learn to recognize them instantly and know the fix for each.

📤
Huge Shuffle Write

What you see: Stages tab → Shuffle Write column shows GBs where you expected MBs.

Root cause: No pre-aggregation before shuffle (no partial HashAggregate), or shuffling unnecessarily wide key, or missing broadcast opportunity on a small join side.

Fix: Check SQL tab for HashAggregate (partial) before Exchange — if absent, Spark couldn't pre-agg. Reduce shuffle key columns. Consider broadcasting the smaller side.
🐌
One Task Running Much Longer

What you see: Task duration histogram has a long right tail — one or a few tasks take 10–100× longer than median.

Root cause: Data skew. One partition has far more data due to a hot key (Zipf-distributed account_id is the classic LedgerForge case).

Fix: Enable AQE skewJoin optimization. Salt the skewed join key manually. Filter hot keys and process separately with a union.
🗑️
High GC Time

What you see: Stages tab → task GC time > 10% of task time, or Executors tab → GC Time column is high for specific executors.

Root cause: Partitions too large for the executor heap, causing frequent garbage collection. Or large objects captured in task closures.

Fix: Increase partition count to reduce per-partition size. Increase spark.executor.memory. Audit task closures for large captured objects.
🔬
Too Many Tiny Tasks

What you see: Stages tab → 200 tasks, each processing <1MB, completing in 10–50ms. Stage duration dominated by scheduling overhead.

Root cause: spark.sql.shuffle.partitions is too high for your data volume.

Fix: Lower shuffle.partitions to match data. Enable AQE partition coalescing. Use .coalesce(N) after large filters that dramatically reduce volume.
📡
Missing Broadcast Join

What you see: SQL tab shows SortMergeJoin where you expected BroadcastHashJoin for a small dimension table.

Root cause: autoBroadcastJoinThreshold set too low or to -1, or optimizer size estimate is too high (missing ANALYZE TABLE).

Fix: Check autoBroadcastJoinThreshold in Environment tab. Use broadcast() hint explicitly. Run ANALYZE TABLE on the small table. Enable AQE for runtime upgrade.
📁
Missing Partition Pruning

What you see: SQL tab → FileScan node → PartitionFilters: [] is empty even with a filter on a partition column. Stages tab input size matches full dataset.

Root cause: Filter not pushed to file scan level — UDF wrapping the partition column, expression-based filter, or filter applied after a join.

Fix: Use direct comparisons on partition column (not expressions). Avoid UDFs on partition columns. Ensure filter references the partitioned table directly.
⚙️
Codegen Disabled (UDF Detected)

What you see: SQL tab physical plan — operations outside *(N) WholeStageCodegen blocks. DeserializeToObject, MapElements, SerializeFromObject nodes visible.

Root cause: A UDF or Python operation broke the WholeStageCodegen boundary. Everything outside the codegen block is interpreted, not JIT-compiled.

Fix: Replace UDFs with native Spark functions (when, otherwise, regexp_extract, etc.). For unavoidable UDFs, use pandas_udf (Arrow-based vectorized, 10–100× faster than row-by-row).
💾
Disk Spill

What you see: Stages tab → Spill (Disk) column shows non-zero values. Tasks are writing temporary spill files to disk during shuffle.

Root cause: Partition size too large for available executor memory. The in-memory shuffle buffer fills before write can complete.

Fix: Increase spark.sql.shuffle.partitions (more, smaller partitions). Increase spark.executor.memory. Note: with AQE, increase partitions — don't decrease (coalescing won't help spill).
