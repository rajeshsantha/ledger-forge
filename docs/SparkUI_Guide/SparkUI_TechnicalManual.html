SECTION 01
Spark Execution Model
The mental model behind every metric you'll ever see in Spark UI. Without this foundation, every number is noise.
1.1 The Transformation Pipeline

When you write DataFrame code, Spark does not execute it immediately. It builds a tree of transformations, passes it through multiple optimization phases, then executes it lazily when an action is called. Understanding this pipeline is what separates guessing from knowing.

TRANSFORMATION PIPELINE — YOUR CODE TO EXECUTION
Your DataFrame Code← What you write
↓
Unresolved Logical Plan← Parsed AST — column names not yet validated
↓ Analyzer — resolves names against catalog
Resolved Logical Plan← Column types resolved, relations checked
↓ Catalyst Optimizer — rule-based + cost-based rewrites
Optimized Logical Plan← Predicates pushed, projections pruned, joins reordered
↓ Planner → Cost Model → selects cheapest physical plan
Selected Physical Plan← The plan Spark will actually execute
↓ WholeStageCodegen — JIT-compiles hot paths
Code-Generated Physical Plan← Fused bytecode — this is what runs
↓ Action triggers execution → DAG: Job → Stages → Tasks
1.2 Key Concepts
ConceptDefinitionWhat You See in UI
JobCreated every time Spark executes an action (.collect(), .count(), .write, .show()). One action = one Job.A row in the Jobs tab
StageA set of tasks that can run in parallel without a shuffle. Stage boundaries are created wherever an Exchange node appears in the physical plan.A stage in the Stages tab; a color block in the DAG
TaskThe smallest unit of execution. One task processes exactly one partition of data. Task count = partition count for that stage.Each bar in the task duration histogram
PartitionA logical chunk of data. After a shuffle, count is controlled by spark.sql.shuffle.partitions (default 200) unless AQE coalesces them.Number of tasks in a stage
ShuffleRedistributing data across executors by key. Involves shuffle write (map-side, to local disk) and shuffle read (reduce-side, fetch from all map tasks). Expensive.Shuffle Read/Write columns in Stages tab; Exchange node in SQL tab
DAGDirected Acyclic Graph of stage dependencies. Shows which stages must complete before others can start.Visual graph in the SQL tab; arrows in Stages tab
1.3 Catalyst Optimizer

Catalyst is Spark's query optimization framework. It applies transformations in two passes before execution begins:

Rule-Based Optimization (RBO)
Always applied regardless of statistics. Includes: predicate pushdown (filters close to scan), projection pruning (read only referenced columns), constant folding (1+1 → 2), and boolean simplification. These are deterministic improvements.
Cost-Based Optimization (CBO)
Applied only if table statistics are available via ANALYZE TABLE. Enables join reordering (smaller table on build side) and join strategy selection based on estimated row counts and data sizes.
💡 Inspect all four plan levels by calling .explain(true) or .explain("formatted") on any DataFrame. The SQL tab shows the final executed plan interactively.
1.4 Adaptive Query Execution (AQE)

AQE is Spark's runtime optimizer (Spark 3.0+, enabled by default in 3.2+). Unlike Catalyst, which optimizes before execution, AQE re-optimizes the plan between shuffle stages based on actual runtime statistics — not estimates.

🔀 Partition Coalescing
Merges tiny post-shuffle partitions into larger ones, reducing scheduling overhead from empty/small tasks.
⚖️ Skew Join Optimization
Detects skewed partitions at runtime and splits them into multiple sub-tasks, eliminating stragglers without manual salting.
📡 Dynamic Broadcast Join
Upgrades a SortMergeJoin to BroadcastHashJoin at runtime if the actual build side is smaller than the broadcast threshold — even if the estimate was wrong.
🔍 In UI: AdaptiveSparkPlan
When AQE is active, the SQL tab wraps the entire plan in AdaptiveSparkPlan. Always read the final executed plan — the initial plan may not reflect what Spark actually did.

2.3 SQL / DataFrame Tab

The most important tab for performance debugging. Every DataFrame query plan is rendered as an interactive DAG with per-node row counts, timing, and data sizes. This is where you find exchange counts, join strategies, and filter pushdown status.

What to Find: Exchange Nodes
Exchange nodes = shuffles. Count them. Each one is a stage boundary and a network round trip. Your primary optimization target.
What to Find: BroadcastExchange
Confirms a broadcast join. Should appear on the small side only. If it appears on a large table, you have a driver OOM risk.
What to Find: FileScan Filters
PartitionFilters = directory-level partition pruning. PushedFilters = row-level predicate pushdown inside the file reader. Both should be populated for efficient scans.
What to Find: Row Count Anomalies
A row count explosion between nodes = unexpected cross join or missing filter. A row count collapse = heavy filtering — confirm it happens early, close to the Scan node.

SECTION 03
How to Read the DAG
The DAG in the SQL tab is read bottom to top — data flows upward. Each node represents one physical operation.
EXAMPLE DAG — AGGREGATION PIPELINE (READ BOTTOM TO TOP)
Result to Driver← Collected by action
↑
HashAggregate (Final)← Merges partial aggregates after shuffle
↑
Exchange hashpartitioning← Stage boundary — shuffle happens here
↑
HashAggregate (Partial)← Pre-shuffle partial agg reduces shuffle volume
↑
Filter← Applied after scan if not pushed down
↑
FileScan (Parquet)← Entry point — look for PartitionFilters & PushedFilters here
3.1 Node Reference — Every Physical Plan Node
FileScan / Scan I/O
Reads data from external storage. The most important fields are PartitionFilters (directory-level, for partitioned datasets — huge speedup) and PushedFilters (row-level, inside the file reader). If your WHERE clause filter doesn't appear in PushedFilters, data is being read in full then filtered — very wasteful.
Exchange SHUFFLE
The most important node to minimize. Three types: hashpartitioning(key, N) for joins/groupBy; SinglePartition for orderBy on large data (dangerous — collects everything to one task); BroadcastExchange for small-table broadcast (desirable).
HashAggregate COMPUTE
Two-phase aggregation. The pre-shuffle partial agg dramatically reduces shuffle volume by computing local partial results first. If you see only one HashAggregate (no partial), Spark couldn't pre-aggregate — check for DISTINCT semantics blocking it.
SortMergeJoin JOIN
Default large-large join. Requires two Exchange nodes (one per side) plus Sort nodes before the merge. Expensive but memory-safe. When you see this with a small table on one side, broadcast is likely a better choice.
BroadcastHashJoin JOIN
Zero shuffle for the large side. Small table is collected by driver, broadcast to all executors as a hash table, large side probed locally. The fastest join strategy. Always look for this when one side is small.
Window COMPUTE
Window function computation (row_number(), rank(), lag()). Requires an Exchange (shuffle by PARTITION BY key) and a Sort within each partition. Expensive — verify the PARTITION BY key has good cardinality.
Sort COMPUTE
Sorts rows within a partition. Appears before SortMergeJoin (both inputs must be sorted) and before Window functions. Also appears when you call .orderBy() — a global sort on large data requires an Exchange SinglePartition which is catastrophic.
Filter / Project TRANSFORM
Filter: boolean predicate on rows. If appearing above a FileScan instead of pushed into it, data is read in full then filtered. Project: column selection — normally implemented as column pruning in columnar formats (Parquet only reads referenced columns).
3.2 WholeStageCodegen

Marked with an asterisk *(N) in the explain output. Everything inside this wrapper node has been compiled into a single JVM bytecode method — Spark fuses multiple operations into tight loops, eliminating virtual dispatch and object allocation overhead. This is why native expressions outperform equivalent UDFs by 3–10×.

✅ Inside Codegen
*(1) Project [a, b, CASE WHEN...] — native expressions, when/otherwise, built-in functions. All fused into one tight generated loop. Best performance.
❌ Breaks Codegen
DeserializeToObject → MapElements → SerializeFromObject — UDFs break the codegen boundary. Each row crosses the Spark→JVM boundary with boxing/unboxing. Every UDF is a performance regression.

4.4 Shuffle Spill

When a shuffle task's in-memory buffer fills up, Spark spills to a temp file on disk. This adds extra I/O: write spill + read spill + write final output — three disk passes instead of one.

MetricMeaningAction
Spill (Memory)Amount of data in memory before spilling — the "would-be" in-memory size.High = partitions are too large for available heap
Spill (Disk)Amount written to the spill temp file. Should be 0 in a well-tuned job.Increase shuffle.partitions to reduce partition size; or increase executor.memory
💡 Spill is almost always fixed by increasing spark.sql.shuffle.partitions (more, smaller partitions) or increasing spark.executor.memory. If AQE is enabled, enabling coalescing won't help spill — coalescing reduces partition count which makes spill worse. Increase partitions first.
4.5 — 4.6 Compression & Bottleneck Detection

Spark compresses shuffle data by default (spark.shuffle.compress = true, using LZ4). This reduces network and disk I/O at the cost of CPU. For CPU-bound jobs on fast internal networks, consider disabling compression.

SHUFFLE BOTTLENECK CHECKLIST — SQL TAB
1. Count Exchange nodes — ask: is each Exchange necessary?
2. Can any be eliminated by broadcasting, pre-partitioning, or restructuring?
3. Check shuffle read size per task in Stages tab — imbalance = skew
4. Check Spill (Disk) in Stages tab — 0 = healthy
5. Is partial HashAggregate present before each Exchange? — yes = good
SECTION 05
Join Strategies
Choosing the right join strategy is one of the highest-leverage Spark optimizations. Understanding what each strategy looks like in the UI lets you verify your optimizer is making good decisions.
BroadcastHashJoin (BHJ)
⚡ FASTEST

When Spark uses it: Small side < autoBroadcastJoinThreshold (default 10MB). AQE can upgrade to BHJ at runtime even if the estimate was wrong.

Mechanism: Driver collects small side → broadcasts hash table to all executors → large side scanned locally and probed. Zero shuffle for large side.

In UI: BroadcastHashJoin + BroadcastExchange (small side only). No Exchange for the large side. Fewer stages than SMJ.

Risk: Broadcast table too large → driver OOM. Watch Size in Memory on driver executor.

import org.apache.spark.sql.functions.broadcast
largeDF.join(broadcast(smallDF), "key")
SortMergeJoin (SMJ)
🐢 DEFAULT

When Spark uses it: Both tables are large (neither qualifies for broadcast). Default for large-large joins.

Mechanism: Both sides shuffled by join key (two Exchanges) → both sides sorted within partitions → merge step walks both sorted streams simultaneously.

In UI: SortMergeJoin with two Exchange hashpartitioning nodes as inputs, each fed by a Sort. 3 stages minimum.

Optimization: If one side can be pre-partitioned (bucketed), one Exchange can be eliminated. If one side is small enough at runtime, AQE upgrades to BHJ.

ShuffleHashJoin (SHJ)
⚡ FAST

When Spark uses it: Manually enabled via spark.sql.join.preferSortMergeJoin = false. One side is smaller but too large to broadcast.

Mechanism: Shuffles both sides, builds a hash table from the smaller side. No sorting required. Faster than SMJ for moderately sized tables.

Risk: Can OOM if hash table doesn't fit in memory. Spark 3.x prefers SMJ by default for stability. Only enable explicitly when you know your data size.

CartesianProduct
💀 DANGEROUS

When Spark uses it: A join with no join condition (cross join) or when join conditions are written incorrectly — e.g., using the wrong DataFrame reference.

Performance: O(M × N) output rows. Catastrophically expensive for large inputs. 50M × 5M = 250 trillion rows attempted.

In UI: CartesianProduct node with a dramatically higher row count output than either input. If you see this unexpectedly, your join condition references the wrong table or is missing entirely.

5.5 Join Strategy Decision Tree
Q: Is the small side < autoBroadcastJoinThreshold (10MB)?
YES → BroadcastHashJoin ← fastest, zero shuffle for large side
NO →
Q: Is preferSortMergeJoin = false AND small side fits in hash table?
YES → ShuffleHashJoin ← faster than SMJ, no sort required
NO → SortMergeJoin ← default safe choice, two shuffles + sorts
AQE override: even if SMJ is selected at plan time, AQE may upgrade to BHJ at runtime if the actual build side is smaller than the threshold after filtering.
SECTION 06
Stage-Level Debugging
The Stages tab is where you diagnose the root cause of slowness. Data skew, GC pressure, spill, and network bottlenecks all manifest here.
6.1 Detecting Data Skew

Click into a Stage and look at the task duration histogram. Skew manifests as a long right tail — median vs. max task time divergence is the clearest signal.

✅ Healthy Stage
Narrow histogram — nearly all tasks complete in similar time. Median ≈ P95 ≈ Max. Shuffle read size is uniform across tasks.
❌ Skewed Stage
Long right tail — 199 tasks finish in 2s, 1 task takes 4 minutes. Sort tasks by Shuffle Read Size descending to confirm one task reads far more data than others.
MetricHealthy RangeProblem Signal
Task duration spreadNarrow histogram (bell-shaped)Long right tail = skew
GC time / Task time< 5%> 10% = heap pressure
Spill (Disk)0 bytesAny value = OOM pressure
Shuffle Read Size (max/median)Ratio ≈ 1–3×Ratio > 10× = hot key confirmed
Failed Tasks0Any failures = instability
Fetch Wait Time< 1 secondHigh = network bottleneck or slow shuffle service
6.2 Straggler Root Causes
Data Skew
Task's partition has far more data than others. Fix: AQE skew join optimization, manual salting, or filter hot keys and process separately with a union.
GC Pressure
Task JVM heap is full, causing long GC pauses. Fix: increase partition count (smaller partitions), increase executor.memory, check for closures capturing large objects.
Disk Spill
Partition is too large for available executor memory, writing to disk. Fix: increase shuffle.partitions to reduce partition size, or increase executor.memory.
Remote Shuffle Fetch
Task waiting for shuffle blocks from slow nodes. Fix: check Fetch Wait Time — if high, investigate network topology or enable external shuffle service.
SECTION 07
Adaptive Query Execution
AQE is enabled by default in Spark 3.2+. Understanding its three optimizations and how they appear in the UI is essential for modern Spark debugging.
7.1 Coalescing Shuffle Partitions

Problem: You set shuffle.partitions = 200, but after a 99%-selective filter, the remaining 200 partitions are 1KB each. 200 tiny tasks — 99% scheduling overhead, 1% actual work.

AQE solution: After each shuffle completes, AQE inspects actual partition sizes and merges small adjacent partitions into larger ones automatically.

In the UI: SQL plan shows AQEShuffleRead instead of Exchange. Task count in the post-shuffle stage will be far less than shuffle.partitions (e.g., 200 → 15 tasks).

spark.sql.adaptive.coalescePartitions.enabled = true
spark.sql.adaptive.coalescePartitions.minPartitionSize = 1MB  ← lower = more aggressive coalescing
7.2 Skew Join Optimization

Problem: One partition in a SortMergeJoin is 50GB while others are 50MB. The task processing that partition runs for 30 minutes, holding up the entire stage.

AQE solution: Detects skewed partitions (larger than skewedPartitionFactor × median AND larger than skewedPartitionThresholdInBytes) and splits them into multiple sub-tasks. The matching partition from the other side is read multiple times.

In the UI: SQL plan shows SkewJoin=true annotation on the SortMergeJoin node. The skewed stage has more tasks than shuffle.partitions because the skewed partitions were split.

spark.sql.adaptive.skewJoin.enabled = true
spark.sql.adaptive.skewJoin.skewedPartitionFactor = 5  ← partition is skewed if size > 5× median
spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes = 256MB
7.3 Dynamic Broadcast Join

Problem: The optimizer estimates a table as 500MB (too large to broadcast) but the actual data after filtering at runtime is only 5MB. Static planning chose SMJ — 2 expensive shuffles — for a join that could have been BHJ.

AQE solution: After the build side of a join shuffles, AQE measures the actual size. If it's below the broadcast threshold, AQE converts the SortMergeJoin to a BroadcastHashJoin at runtime.

In the UI: The physical plan initially shows SortMergeJoin, but after execution the plan is updated to show BroadcastHashJoin. You'll see AdaptiveSparkPlan at the root with a "re-optimized plan" section showing the runtime decision.

💡 Always read the final executed plan in the SQL tab, not the initial plan. With AQE active, the initial plan is a best-guess that may be significantly different from what was actually executed.
SECTION 08
Performance Debugging Playbook
Symptom → root cause → fix. A repeatable diagnostic workflow for every Spark performance problem you'll encounter.
🐌
Symptom: Slow Join

1. Open SQL tab → find the query → look at the join node type. Is it SortMergeJoin when one side is small? → Force broadcast with broadcast() hint.

2. Is it BroadcastHashJoin but still slow? → The broadcast table may be too large, causing driver OOM. Check driver memory usage.

3. Count Exchange nodes → can any join be eliminated by pre-partitioning (bucketing) on the join key?

4. Check AQE — if enabled, did it upgrade to BHJ at runtime? Look for AdaptiveSparkPlan and inspect the final plan.

⚖️
Symptom: Skewed Key (Straggler Task)

1. Stages tab → find the long-running stage → click into it → task duration histogram → identify the straggler count.

2. Sort tasks by Shuffle Read Size descending → confirm the straggler reads far more data (100× typical).

3. Solutions in order of preference: (a) Enable AQE skew join optimization — handles most cases automatically. (b) Manual salting: add random integer 0–N to the key, duplicate the other side N times, join on (key, salt). (c) Filter out the hot key and process separately with a union.

🔀
Symptom: Too Many Shuffle Partitions

1. SQL tab → count Exchange nodes. After each Exchange, check the stage's task count and per-task data size in Stages tab.

2. If tasks are processing <1MB each → too many partitions → reduce spark.sql.shuffle.partitions or enable AQE coalescing.

3. If AQE is enabled and partitions are still tiny → check minPartitionSize config — it may be too high, preventing coalescing.

Formula: Optimal partition count ≈ Total shuffle write / 128MB. Example: 100GB shuffle → ~800 partitions.

📡
Symptom: Missing Broadcast Join

1. SQL tab shows SortMergeJoin where you expected BroadcastHashJoin.

2. Check spark.sql.autoBroadcastJoinThreshold in Environment tab — if -1, broadcasts are disabled.

3. If threshold is set but BHJ isn't happening: optimizer's size estimate is too high. Run ANALYZE TABLE to update statistics, or force with broadcast() hint.

4. If AQE is enabled and still no BHJ: the actual runtime size may genuinely exceed the threshold.

📁
Symptom: Missing Partition Pruning

1. SQL tab → click the FileScan node → look for PartitionFilters — if empty, your filter was not pushed to directory-level scan.

2. Common causes: filtering via a UDF (Catalyst can't push opaque functions), wrapping partition column in an expression (year(event_date) = 2024 won't prune — use event_date >= '2024-01-01' instead), or filtering after a join (filter must reference the partitioned table directly).

3. Note: PushedFilters (row-level, inside Parquet reader) is different from PartitionFilters (directory-level). Both should be populated for maximum efficiency.

🗑️
Symptom: High GC / Executor Memory Pressure

1. Executors tab → check GC Time column → if >10% of Task Time on specific executors, those executors hold oversized partitions.

2. Solutions: increase partition count to reduce per-partition size; increase spark.executor.memory; tune spark.memory.fraction and spark.memory.storageFraction; check for closures capturing large objects in transformations.

⚙️
Symptom: Codegen Disabled (UDF in Plan)

1. SQL tab physical plan → operations appear outside *(N) WholeStageCodegen blocks. You see DeserializeToObject, MapElements, SerializeFromObject nodes.

2. What it means: a UDF broke the WholeStageCodegen boundary. Everything outside the codegen block is interpreted, not JIT-compiled — 3–10× slower.

3. Fix: replace UDFs with native Spark functions (when, otherwise, regexp_extract, concat, etc.). For Scala UDFs that can't be eliminated, consider vectorized pandas_udf (Arrow-based, processes batches).

SECTION 10
How Senior Engineers Use Spark UI
A senior Spark engineer never looks at Spark UI randomly. They follow a deliberate workflow — systematic, hypothesis-driven, one change at a time.
1
Establish the Baseline
Before any optimization, run the job and record: total job duration, stage durations, key exchange sizes, task count per stage, and any spill. Screenshot or log these numbers. Without a baseline, you can't measure improvement — you're just guessing.
2
Start With the SQL Tab
Find the query for the slow operation. Read the physical plan as text (also call .explain("formatted") in code for better readability). Count Exchange nodes. Identify join types. Check for expected optimizations: BHJ, partition filters, codegen markers (*(N)). Form a hypothesis about what's wrong before looking at stage metrics.
3
Find the Bottleneck Stage
Go to Stages tab. Sort by Duration descending. Click into the slowest stage. Look at the task duration histogram. Ask three questions: Is it data skew (right tail)? Is it GC (check GC Time column)? Is it spill (check Spill Disk column)? The histogram tells you the root cause — not just that it's slow, but why.
4
Correlate Stage to Plan
Once you've found the slow stage, go back to the SQL tab and find which physical plan node corresponds to that stage. This correlation is the key insight: now you know exactly which operation is slow (a join, a window function, an aggregation) and why (skew, wrong join strategy, missing filter pushdown).
5
Check Executors for Systemic Issues
Executors tab: are all executors healthy? Is GC high on specific executors? Is memory pressure causing cache evictions? Are some executors processing far more data than others? Systemic issues (bad node, OOM, network partition) won't show in the stage view alone.
6
Apply One Change at a Time
Never apply multiple optimizations simultaneously — you won't know which one helped. Apply the most impactful change (e.g., enable AQE, force broadcast, increase shuffle partitions) and re-run. If you apply 3 changes and get 2× speedup, you learned nothing about why it improved or which change to keep.
7
Compare Plans, Not Just Times
After optimization, go to the SQL tab and confirm the plan changed as expected. A 2× speedup from a plan that's structurally wrong is luck — you got faster data or a favorable partition distribution. A plan that's structurally correct (fewer exchanges, BHJ where expected, pruning active) will scale reliably to larger data.
8
Validate at Scale
A local test with 1M rows proves nothing. Re-run the optimized job on production data (or a representative sample at full scale) and verify the improvements hold. Some optimizations that work at small scale break at large scale — especially broadcast join thresholds and AQE configurations.
THE MENTAL MODEL

Every second of job time is spent doing one of: reading data, computing (CPU), writing shuffle, waiting for network, or waiting for GC.

The Spark UI makes all of these visible. The job of performance engineering is to systematically eliminate the biggest bottleneck, then find the next one. Don't optimize the second-biggest bottleneck until you've handled the first.
